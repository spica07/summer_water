# 물놀이장 데이터 조사·업데이트

전국 17개 시·도, 229개 시·군·구가 이미 등록돼 있다(서울 25개 자치구는 별도 xlsx 파이프라인 소관, 나머지 204개는 이 스킬로 관리). 인자(`$ARGUMENTS`)로 조사 범위를 받는다:
- 지역명 (예: `경기`, `부산`, `제주`, `강원`) — 해당 시·도 전체 지자체 재조사
- 지자체명 (예: `고양시`, `인천 서구`, `강원 고성군`) — 해당 지자체 하나만 재조사
- `재확인` — 현재 데이터에서 status가 "전년도 확인"/"재확인 필요"인 지자체만 전국 대상으로 재조사 (assets/js/data.js의 FACILITIES에서 해당 status의 district 목록 추출)
- `전국` 또는 인자 없이 광범위 갱신 요청 — 17개 시·도 전체를 재조사 (배치 수가 매우 많아지므로 실행 전 사용자에게 범위·소요 시간을 확인)
- 인자가 모호하면 사용자에게 범위를 물어본다.

## 절차

1. **배치 구성**: 조사 대상 지자체를 5~8개씩 배치로 묶는다.
   - 광역시(부산·대구·인천·광주·대전·울산)·세종·제주는 보통 1개 시·도 = 1배치로 충분.
   - 경기(31개)·경북(23개)·전남(22개)·경남(18개)·강원(18개)·충남(15개)·전북(14개) 등 지자체 수가 많은 도는 지리적으로 2~6개 배치로 나눈다 (예: "경기 대도시권/서남부/북동부", "강원 영서/영동", "경북 북부/남부", "전남 동부/서부"). 배치당 지자체 7~16개, 웹 검색이 지자체당 2~6회씩 필요하므로 배치가 너무 크면 에이전트 시간이 오래 걸리고 세션 한도에 걸릴 수 있다.
   - 각 배치를 `water-researcher` 서브에이전트(Agent 도구, subagent_type: "water-researcher")로 **한 메시지에서 병렬 실행**한다. 프롬프트에는 대상 지자체 목록과 region 값만 쓰면 된다 — 조사 규칙·municipality 표기 규칙(광역시 접두사, 고성군 등 동명 지자체 구분)·출력 스키마는 에이전트 시스템 프롬프트에 내장돼 있다.
   - 주의: Workflow 도구는 쓰지 않는다(사용자가 대규모 오케스트레이션을 원치 않음). 일반 Agent 병렬 실행을 사용.
   - 에이전트가 세션 한도 등으로 실패하거나 결과 없이 끝나면(`result`가 빈 텍스트/에러) 같은 배치를 재실행한다. 결과가 중간에 멈춘 것 같으면 SendMessage로 해당 agentId에 "지금까지 조사한 것만으로 즉시 최종 JSON 제출" 요청을 보내 마무리시킬 수 있다.

2. **결과 수집**: 각 에이전트의 최종 JSON을 스크래치패드에 `research_<배치명>.json`으로 저장한다.
   - 주의: 에이전트 .output 파일이 0바이트인 경우가 있다. 완료 알림(result)에 담긴 JSON 텍스트를 Write로 직접 파일화하는 것이 확실하다.

3. **병합**: `py -3 tools/merge_extra.py <저장한 파일들...>` 실행.
   - 기존 `tools/facilities_extra.json`을 기준으로, 입력에 등장한 지자체는 **통째로 교체**(업데이트), 새 지자체는 추가된다. facilities가 빈 배열이면 해당 지자체 항목이 제거되므로, 조사 실패와 "시설 없음"을 구분해서 넣을 것.
   - 서울 데이터는 xlsx 파이프라인(extract.py) 소관이므로 이 스킬로 건드리지 않는다.

4. **재생성**: `py -3 tools/geocode.py`를 **백그라운드로** 실행 (Nominatim 1.1초/쿼리 제한 — 신규 시설이 많으면 10분 이상). `python` 명령은 Windows 스토어 스텁이라 실패하니 반드시 `py -3`.

5. **코드 체크리스트**: 17개 시·도, 229개 시·군·구가 이미 `tools/geocode.py`(`REGION_BOX`, `XXX_CENTERS`, `DISTRICT_CENTERS`/`DISTRICT_REGION`)와 `assets/js/app.js`(`REGION_ORDER`, `REGION_VIEW`), `assets/js/report.js`(지역 타일 배열)에 전부 등록돼 있으므로 **일반적인 재조사·시설 추가 작업에서는 코드 수정이 필요 없다**. 아래 경우에만 코드를 손봐야 한다:
   - 행정구역 자체가 바뀌는 경우(시·군 통합/분리, 새 특별자치시 지정 등) — `DISTRICT_CENTERS`/`DISTRICT_REGION`에 신규 지자체 좌표 추가, 필요시 `REGION_ORDER`/`REGION_VIEW`도 갱신.
   - `merge_extra.py` 실행 시 "알 수 없는 지자체" 경고가 뜨는 경우 — municipality 표기가 `DISTRICT_CENTERS`의 키와 정확히 일치하는지 먼저 확인(오타·접두사 누락 여부)하고, 그래도 신규 지자체라면 위와 동일하게 추가.

6. **검증**: data.js 재생성 후 —
   - `node --check assets/js/*.js`
   - FACILITIES 총계·지역별 수·중복 id·좌표 범위·status/type 어휘 검사 (스크래치패드에 검증 스크립트 작성)
   - 결과 요약(지역별 시설 수, A/B/C/D 등급 분포, 교체된 지자체 목록)을 사용자에게 보고

7. 커밋/푸시는 사용자가 요청할 때만 한다.
